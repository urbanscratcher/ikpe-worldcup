# Development Notes

이 문서는 구현 중 생기는 질문, 고민, 결정, 학습 포인트를 짧게 기록한다.

## 기록 원칙

- 구현 방법 자체보다 "왜 그렇게 했는지"를 남긴다.
- 나중에 다시 볼 가치가 있는 질문과 결정을 우선한다.
- 확정되지 않은 생각은 `질문` 또는 `고민`으로 남기고, 결정되면 `결정`을 추가한다.
- 코드 변경 이력은 git에 맡기고, 이 문서는 맥락을 보존한다.

## 기록 형식

새 항목은 아래 형식을 기본으로 쓴다.

```md
### 질문: 제목

- 상황:
- 고민:
- 선택지:
- 현재 결론:
- 나중에 확인할 것:
```

결정이 끝난 항목은 아래처럼 남긴다.

```md
### 결정: 제목

- 결정:
- 이유:
- 포기한 것:
- 다시 볼 조건:
```

학습 포인트는 아래처럼 짧게 남긴다.

```md
### 배운 점: 제목

- 알게 된 것:
- 왜 중요한가:
- 다음에 적용할 방식:
```

## 에이전트 역할

- 사용자가 구현 중 던진 질문과 고민을 이 문서에 축적한다.
- 코드 작성을 먼저 제안하지 않고, 사용자가 직접 구현할 수 있도록 맥락과 선택지를 정리한다.
- 결정이 바뀌면 이전 항목을 지우지 않고 새 결정으로 덧붙인다.
- 구현 순서와 현재 기준 문서가 바뀌면 링크를 갱신한다.

## 2026-07-10

### 결정: 시간 제약 때문에 첫 실행 뼈대는 에이전트가 구현한다

- 결정: Vite React 웹 앱, Hono Worker API, shared Valibot schema를 먼저 연결해 눈으로 확인 가능한 입장 코드 흐름을 만든다.
- 이유: 행사까지 시간이 짧아서 프로젝트 구조, 실행 스크립트, API 연결, httpOnly 쿠키 흐름을 먼저 안정화해야 한다.
- 포기한 것: Supabase 실제 DB 연결, 관리자 CRUD, 토너먼트 생성은 이번 첫 뼈대에 넣지 않는다.
- 다시 볼 조건: 입장 코드 mock 흐름을 화면에서 확인한 뒤 Supabase `audience_codes` 검증으로 교체한다.

### 배운 점: Wrangler compatibility date는 설치된 workerd가 지원하는 날짜 이하여야 한다

- 알게 된 것: `compatibility_date`를 현재 날짜로 잡아도 로컬 `workerd` 바이너리가 그 날짜를 아직 지원하지 않으면 서버가 시작되지 않는다.
- 왜 중요한가: Worker 프로젝트를 처음 만들 때 오늘 날짜를 무작정 넣으면 로컬 개발이 막힐 수 있다.
- 다음에 적용할 방식: Wrangler가 지원하는 최신 날짜로 두고, 의존성을 업그레이드할 때 compatibility date도 같이 올린다.
